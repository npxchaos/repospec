# Forge Protocol — Specification v0.1

*Working draft. This defines what actually lives inside a `.forge/` directory and how it behaves.*

> **Naming note.** `forge` collides with Foundry's Solidity toolchain, whose literal first command is `forge init`. Treat every `forge` command below as a placeholder pending a rename — the protocol design doesn't depend on the name.

---

## 0. What Forge is, and what it deliberately isn't

Forge is a repository protocol that makes a codebase self-describing to AI coding agents. It is not an agent, an IDE, or a model.

Two standards already cover part of this ground, and Forge is designed to sit **on top of** them rather than compete:

- **AGENTS.md** is the file coding agents already read for project context. Forge does not invent a rival file. Forge *compiles* AGENTS.md as a build output, so it works with today's tools on day one.
- **GitHub spec-kit** covers the spec → plan → tasks → implement flow for building a single feature. Forge is not feature-scoped. It governs the standing engineering system of the repository — the roles, the durable decisions, the history — that persist across every feature.

Forge contributes two things those standards don't:

1. **Explicit roles** — named responsibilities with owned scope and hard boundaries, replacing "act like a senior engineer."
2. **Durable project memory** — versioned decisions and history that outlive any single conversation or model.

Everything else here — a constitution, a bootstrap interview, a lifecycle — Forge adopts from prior art on purpose, and says so where it does.

## 1. Directory layout

```text
.forge/
├── project.yaml        # identity: what, who, stack, deploy targets, active roles
├── constitution.md     # non-negotiable rules (compiled to the top of AGENTS.md)
├── workflow.md         # the lifecycle and its gates
├── agents/             # role definitions — one file per role
│   ├── architect.md
│   ├── builder.md
│   ├── reviewer.md
│   └── …
├── rules/              # topic rules, referenced by roles and the constitution
│   ├── testing.md
│   ├── security.md
│   └── style.md
├── decisions/          # ADRs — append-only, one file per decision
│   └── 0001-postgres-over-dynamo.md
├── history/            # narrative log of significant repo-level events
│   └── CHANGELOG.forge.md
└── templates/          # scaffolds the roles use

AGENTS.md               # GENERATED by forge — do not hand-edit
```

**Invariant:** humans edit `.forge/`; Forge compiles `AGENTS.md`. The generated file opens with a header marking it generated and pointing back to the source.

## 2. `project.yaml` — identity

```yaml
name: acme-billing
summary: Usage-based billing service for the Acme platform.
users: Internal finance team plus the public billing API.
stack:
  language: TypeScript
  runtime: Node 20
  framework: NestJS
  datastore: PostgreSQL
deploy:
  target: AWS ECS
  pipeline: GitHub Actions
compliance: [SOC2, PCI-DSS]
roles: [architect, builder, reviewer, tester, security, release]
```

`roles` lists which files in `agents/` are active. A project enables only the roles it needs; the rest don't exist for that repo.

---

## 3. `constitution.md` — non-negotiables

*(Concept borrowed from spec-kit, acknowledged.)* A short, stable set of principles every change obeys. Kept deliberately small and compiled verbatim into the top of AGENTS.md, because current agents weight what they read first. Examples:

- Small steps over large rewrites.
- Tests precede implementation for any code touching billing math.
- A human approves every change to `deploy/` or `migrations/`.
- Secrets never enter the repo, the logs, or an agent's output.

---

## 4. `agents/` — roles (the core of Forge)

A role is a **contract, not a personality.** Each file is frontmatter plus a body.

```yaml
---
role: reviewer
owns: review of every diff before merge
authority: may block a merge; may not modify source
reads: [constitution.md, rules/style.md, rules/testing.md]
hands_off_to: tester
escalates_to: human
---
```

Body sections:

- **Responsibilities** — what this role does, concretely.
- **Boundaries** — what it must *not* do (the part that actually constrains an agent).
- **Checklist** — the pass it applies every time.
- **Hand-off** — who it passes work to, and on what signal.

An agent operating "as the reviewer" loads exactly this file: one responsibility, bounded authority, a defined hand-off. Roles chain along the workflow rather than one agent improvising the whole job.

---

## 5. `workflow.md` — the lifecycle as gates

```text
Plan → Design → Implement → Review → Validate → Document → Release → Sync
```

Each step declares its **owning role**, its **entry/exit criteria**, and whether it is a **human gate**. The workflow is a state machine, not a suggestion — a step can't be skipped, and a human gate can't be auto-cleared.

*(The shape generalizes spec-kit's Specify → Plan → Tasks → Implement; Forge extends it across the full lifecycle and binds each step to a role.)*

---

## 6. `decisions/` — durable memory (the other core of Forge)

Append-only, numbered ADRs. This is precisely what AGENTS.md and spec-kit don't persist: *why* the system is the way it is.

```markdown
# 0001 — PostgreSQL over DynamoDB

- Status: accepted (2026-02-14)
- Context: billing needs multi-row transactional integrity.
- Decision: PostgreSQL on RDS as the system of record.
- Alternatives rejected: DynamoDB (no cross-row transactions at the time).
- Consequences: relational migrations; no infinite-scale write path.
- Superseded by: —
```

Agents read `decisions/` to avoid re-litigating settled choices. Superseding a decision links forward; nothing is deleted.

---

## 7. `history/` — what changed

An append-only, human- and AI-readable narrative of significant repo-level events: architecture shifts, role changes, decisions accepted. Distinct from `git log` — it records the evolution of the engineering *system*, not every commit.

---

## 8. Compilation to AGENTS.md

`forge build` compiles `.forge/` into a root `AGENTS.md`:

- constitution → top of file (read first)
- active roles → summarized
- rules → linked, not inlined (progressive disclosure keeps the file lean)
- pointers into `decisions/` and `history/`

This is the adoption move that matters: the artifact agents already consume is Forge's *output*, so the protocol works with current tools immediately, while `.forge/` stays the human-authored source of truth.

---

## 9. Bootstrap

`forge init` interviews the developer — what you're building, who it's for, stack, deploy, standards, compliance, which roles — and generates an initial `.forge/` plus the first `AGENTS.md`. Re-runnable and non-destructive.

---

## 10. Open questions (the honest gaps)

- **Enforcement.** AGENTS.md is advisory; an agent can ignore it. Forge needs a verification story — at minimum a CI check that the compiled AGENTS.md is current, ideally optional gate hooks.
- **Drift.** When code and `.forge/` diverge, which wins, and how is the divergence detected before it misleads an agent?
- **Naming.** See the note at the top.
